# INSTRUCTIONS: PHASED REVIEW
Execute an autonomous review across all files, focusing on the following pillars (1–5 are core; 6–10 are extended use cases for production-style and AI-assisted codebases):

## 1. Contextual Integrity & State
- Check if generated functions in one file correctly interface with existing types in `types/` or global states (e.g., Redux/Zustand).
- Trace cross-module contracts: shared schemas, DTOs, or domain models (e.g., `types/`, OpenAPI, Pydantic/Zod) — flag mismatches between models and callers.
- For UI-heavy apps, audit framework session or client state (e.g., keys, nested mutation, rerender/reconciliation triggers) for consistency across reruns and tabs where relevant.
- Where configuration or feature flags gate behavior, ensure optional paths fail closed (safe defaults) when values are missing or invalid.
- Identify any "hallucinated" helper functions or imports that don't exist in the local project.

## 2. Logic & Edge Cases
- Analyze all complex conditionals for race conditions or off-by-one errors.
- Review concurrency and ordering: async, threads, bulk/batch iterators, pipelines — including partial failure, retries, and idempotency of side effects.
- Verify that every async operation has a robust try/catch block and appropriate user-facing error handling.
- Check for "happy path" bias: Ensure null/undefined checks are present for all API responses.

## 3. Style & "Local Flavor"
- Refactor any code that deviates from our project standards (e.g., if we use arrow functions, ensure the generated code doesn't use standard function declarations).
- Ensure naming conventions match existing patterns (camelCase vs PascalCase).

## 4. Security Audit
- Search for any hardcoded secrets, API keys, or placeholder strings (e.g., "YOUR_API_KEY").
- Validate that all database queries use parameterized inputs to prevent SQL injection.
- Check for insecure direct object references (IDOR) in API routes.
- **Prompt and output safety** (LLM or generative features): user-controlled strings reaching system prompts; validate or sanitize model output before persistence, rendering, or export.
- **Secrets and PII in logs and telemetry**: ensure events, traces, and debug payloads do not include API keys, tokens, or sensitive personal data.
- **Outbound URLs** (webhooks, handoffs, export callbacks): if URLs can be user-influenced, assess SSRF, open redirects, and unsafe request targets.
- **Operational guardrails**: verify kill switches, circuit breakers, or rate limits are consulted on critical paths, not only defined.
- **Dependency risk**: pinning or lockfile strategy; known-vulnerable or unpinned transitive deps in deployable artifacts.

## 5. Testability
- Evaluate the modularity of the code. If a function is too large to unit test easily, suggest a refactor.
- Generate a summary of missing test coverage for the new logic.
- Cross-reference tests against source modules and critical branches (error paths, authorization denial, bulk partial failure).
- Clarify the role of **eval / offline scripts / shadow runs** vs automated tests: whether they are CI gates, manual baselines, or drift checks — and align with documented release criteria.

## 6. Operational Reliability & Boundaries
- **Timeouts and retries** on external clients (HTTP, LLM, queues); avoid unbounded hangs.
- **Idempotent writes** where jobs can retry (filesystem, object storage, DB upserts).
- **Dead-letter queues, incident hooks, or error sinks**: wired and useful, not silent drops.

## 7. Data Governance & Provenance
- **Retention and deletion**: policies enforced on startup, cron, or admin paths; safe behavior on shared or multi-tenant filesystems.
- **Lineage and audit**: provenance for exports, handoffs, and regulated outputs — can you trace what model, version, and inputs produced an artifact?

## 8. Contracts & Documentation Drift
- **Integration contracts**: compare integration specs or `docs/` API descriptions to code under integration/adapters folders.
- **UX or operator docs vs implementation**: spot-check user-facing copy, flows, and component behavior against `docs/ux` or equivalent.

## 9. Experiments & Shadow / Comparison Paths
- For A/B, shadow, or champion/challenger flows: fair comparison (same inputs where intended), no data leakage between arms, and clear metrics.

## 10. Static Quality & Mechanical Checks
- **Tooling gaps**: note absence of linter, typechecker, or formatter where the stack expects them; optional runs (`pytest`, `ruff`, `mypy`, `eslint`, etc.) when present.
- **Mechanical grep passes** (adapt to language): `TODO|FIXME|XXX`; bare `except` or empty handlers; `eval(`, `pickle`, `subprocess` with `shell=True`; hardcoded `http://` where TLS is required; obvious secret-like patterns.
