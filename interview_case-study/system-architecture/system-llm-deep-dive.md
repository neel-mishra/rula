# System LLM deep dive

This document inventories **where large language models are invoked**, **what each call is for**, **how outputs are constrained and validated**, and **how the repository treats determinism**. It reflects the state of the **Rula** monorepo as of the analysis date (code under `rula-gtm-agent/`).

## Scope: what “the system” includes

| Area | LLM involvement |
|------|-----------------|
| **`rula-gtm-agent/`** | **Yes.** Anthropic Claude and Google Gemini are integrated behind a router; prompts live in Python and optional Business DNA injection. |
| **`business dna/`** | **No model calls.** Markdown corpus is parsed into typed slices (`BusinessContextRegistry`) and injected into prompts as **grounded context** (not generated by an LLM). |
| **`rula-landing-page/`** | **No LLM integration** in `app/` (static/marketing UI). |
| **`interview_case-study/`**, **`skills/`** | Documentation and human/agent guidance only; no runtime LLM API. |

The remainder of this document focuses on **`rula-gtm-agent`**.

---

## Architecture overview

1. **`GenerationRequest`** (`src/providers/base.py`): carries `content_type`, `prompt`, optional `system`, `temperature` (default **0.3**), `max_tokens` (default **2048**).
2. **`ModelRouter`** (`src/providers/router.py`): picks provider from **`TASK_MODEL_MAP`** (per `content_type`), honors `AppConfig` keys (`MODEL_PRIMARY`, `MODEL_FALLBACK`, API key presence), calls **`ClaudeProvider`** or **`GeminiProvider`**, emits **`emit_generation`** telemetry, and on failure tries **cross-provider fallback** (Claude ↔ Gemini) with the **same** request.
3. **Providers**: `ClaudeProvider` uses `anthropic` SDK `messages.create` with **`temperature=request.temperature`**. `GeminiProvider` concatenates `system` + `prompt` and calls `generate_content`; it **does not pass `temperature` to the Gemini API** (see Determinism).

---

## Task routing and model prioritization

`TASK_MODEL_MAP` in `src/providers/router.py` assigns a **preferred provider** per logical task. This is the primary “weighting” of **which model family** handles which surface—not an internal model weight for tokens.

| `content_type` | Preferred provider | Role in product |
|------------------|-------------------|-----------------|
| `email`, `email_v3`, `subject_line` | Claude | Outbound **copy** (high-stakes tone, structure). |
| `discovery_questions`, `discovery_questions_v3` | Claude | **Strategic questions** (GTM strategist persona). |
| `rationale_summary` | Claude | Reserved in router; **no production caller found** in `src/` at inventory time. |
| `value_prop_explanation`, `map_synthesis`, `map_committer`, `threshold_justification`, `econ_scenario` | Gemini | **Analyst/extraction/synthesis-style** tasks; `map_committer` and `value_prop_explanation` are wired; others are **routed in the map but not invoked** from `src/` (placeholders for future pipelines). |

**Reasoning vs content generation (conceptual split):**

- **Content generation (Claude):** prospecting email body/subject/CTA; discovery questions. Prompts enforce **structure**, **JSON shape**, and **banned vocabulary**.
- **Reasoning / explanation / extraction (Gemini in code today):** value-prop **explainability** narrative; MAP **committer** JSON extraction. These lean on **evidence lists** and **JSON-only** instructions rather than open-ended prose.

**Important:** Value-prop **ranking and scores** are **not** LLM outputs. They come from deterministic rules in `src/agents/prospecting/value_prop_scoring.py` (`INDUSTRY_RULES`, `SIZE_BREAKPOINTS`, weights, caps). The LLM only **narrates** the precomputed evidence.

## LLM touchpoints (inventory)

### 1. Prospecting Stage 4 — email + discovery questions

- **Files:** `src/agents/prospecting/generator.py`, prompt builders in `src/providers/prompts.py`.
- **Providers:** Claude (`email_v3`, `discovery_questions_v3`).
- **Inputs:** enriched account (company, contact, health plan, employees, notes), `SegmentContext` from `resolve_segment_context`, company context from `fetch_company_context`, Business DNA slices (`voice`, `claims`, `pillars` for email; `voice`, `product`, `icp` for questions) when registry loads.
- **Outputs:** JSON parsed into `OutreachEmail` and `list[str]` questions; policy validators; optional **repair** generation; else **deterministic templates** (`_deterministic_email_v3`, `_deterministic_questions_v3`).

### 2. Value-prop explainability

- **File:** `src/explainability/value_prop_reasoner.py`.
- **Provider:** Gemini (`value_prop_explanation`).
- **Inputs:** winning `ValuePropMatch`, company/industry/size/plan, **scoring attributions** as bullet evidence, optional runner-up score delta, optional Business DNA `pillars` + `product` block.
- **Output:** Plain text; **specificity heuristic** `_score_specificity` must be ≥ **75** or output is discarded for a **deterministic template** fallback.

### 3. MAP committer resolution

- **File:** `src/agents/verification/committer_resolver.py`.
- **Provider:** Gemini (`map_committer`).
- **Inputs:** `evidence_text`, optional `company_profile_markdown` from `build_company_profile_text` (`src/agents/verification/account_profile.py`).
- **Output:** JSON → `CommitterExtraction`; **merged** with regex **heuristics** (prefer LLM name when confident; fill title from heuristics when missing).

### 4. Audit “judge”

- **Files:** `src/agents/audit/judge.py`, `src/schemas/audit.py`.
- **LLM:** **None today.** Docstrings describe a “heuristic stand-in for GPT-4o judge.” `JudgeResult.judge_model` is `heuristic-judge-v1`.

### 5. Telemetry and resilience (supporting, not generation)

- **Generation telemetry:** `src/telemetry/ux_events.py` → `emit_generation` (provider, duration, errors, fallback).
- **Connector policy:** `src/integrations/connector_policy.py` — `LLM_PROVIDER` timeout default **120s**, retries **0** at policy default (override via env).
- **Circuit breaker:** `src/safety/circuit.py` — documented as **future** use for LLM calls (not wired into providers in the reviewed code).

### 6. Prompt assets folder

- **`prompts/README.md`** states markdown templates are **placeholders** for a modelized pipeline; **live prompts are in Python** (`src/providers/prompts.py` and inline builders).

## Factors and inputs (summary)

| Factor | Used by |
|--------|---------|
| Account fields (company, industry, employees, health plan, notes, contact) | Prospecting prompts, profile block for committer, value-prop explanation |
| Segment + wedge + competitor token (`segment_logic`) | Prospecting v3 prompts (deterministic binding before LLM) |
| External/snippet context (`fetch_company_context`) | Email/question **company_context** string |
| Business DNA slices (`prompt_block`) | Email v3, discovery v3, value-prop explanation (when loaded) |
| Stage 2 scoring attributions | Value-prop explanation only |
| MAP evidence text (+ optional CRM-style profile) | Committer resolver |

**Truth-stack precedence** (from `business_context.py` module docstring): code invariants and deterministic rules **outrank** structured Business DNA slices, which **outrank** LLM-generated text (which must pass validators / fallbacks).

## Determinism and randomness controls

| Mechanism | Detail |
|-----------|--------|
| **Default `temperature`** | `GenerationRequest.temperature` defaults to **0.3** (`base.py`). |
| **Per-call overrides** | Prospecting: email initial **0.4**, questions **0.3**, **repair passes 0.2** (`generator.py`). Value-prop explanation: **0.2**. Committer: **0.1**. |
| **Structured outputs** | JSON-only instructions; parsers strip optional markdown fences before `json.loads`. |
| **Pre-bound variables** | Prompt builders state that placeholders are filled **before** invocation (`email_prompt_v3` docstring). |
| **Gemini API** | `GeminiProvider` **does not** forward `temperature` (or seed) to `generate_content`; determinism for Gemini paths relies on **low-temperature intent in code** but **API sampling is not explicitly controlled** in this provider. |
| **Claude API** | `temperature=request.temperature` is passed through. **No `seed`** is set in the Anthropic call. |
| **`GENERATION_MODE` / `fast_mode`** | Present on `AppConfig` and surfaced in `app.py` metadata; **not referenced** in provider or generator logic for sampling (cosmetic/config flag only today). |
| **Provider unavailable / failure** | Router returns error response; prospecting uses **template fallbacks**; value-prop explainability uses **template**; committer uses **heuristics / nulls**. |

### C. Email body v3 — user prompt template (`email_prompt_v3`)

Runtime fields: `prospect_name`, `company_name`, `company_context`, `segment_label`, `health_plan`, `mapped_value_prop` (spaces for underscores), `similar_competitor`, plus optional Business DNA block for slices **`["voice", "claims", "pillars"]`**.

```
Construct a 3-paragraph email body following this strict deterministic structure:

1. The Pattern Interrupt (Relevance): Reference the following company context: "{company_context}". Connect it to the difficulty of maintaining a high-performing workforce. Do not use "I hope you're well."

2. The Straight Line (The Rula Solution): Introduce Rula as the solution to the engagement gap.
* Constraint: You must mention that Rula is free for the employer and works with {health_plan} to remove financial friction.

Segment Logic: If the segment is a Health System, focus on "Total Cost of Care." If it is a University, focus on "Student/Staff Access".
The account segment is: {segment_label}. The mapped value proposition is: {mapped_value_prop with underscores replaced by spaces}.

3. The Offer (The MAP): Frame the meeting as a discussion about a "Marketing Partnership" rather than a sales pitch.
* The Pitch: "We handle the provider network and the internal marketing campaigns so your team doesn't have to".

The CTA: "Open to seeing the campaign playbook we used to drive utilization at {similar_competitor}?"

Strict Banned List: "Revolutionary," "pioneering," "partnership" (in the first sentence), "help," "excited," "reached out," "checking in."

Additional context:
- Prospect name: {prospect_name}
- Company: {company_name}
- Health plan: {health_plan}

Return EXACTLY this JSON (no markdown fences):
{"subject_line": "...", "body": "...", "cta": "..."}

Rules:
1. Subject line: < 60 chars, mention the company name, no clickbait.
2. Body: exactly 3 paragraphs following the structure above.
3. CTA: use the campaign-playbook framing specified above.
4. Never use words from the Strict Banned List.
5. Only use claims that appear in the Business Context section below. Do not invent statistics.
{optional Business DNA block}
```

**Repair extension** (appended by `generator.py` when validation fails):

```
The previous output had these policy violations. Fix them and return corrected JSON:
- {violation 1}
- {violation 2}
...
```

## How prompts are structured for reliable output

1. **Role + task separation:** `system` sets persona; `prompt` carries procedural rules and variable data (Claude). Gemini often receives a merged string but the same conceptual split is preserved in code.
2. **Explicit output contract:** JSON shape spelled in text; “no markdown fences” to reduce parsing failures.
3. **Deterministic scaffolding inside the user prompt:** Numbered paragraphs, banned lists, segment logic branches—reduces drift vs freeform “write an email.”
4. **Grounding hooks:** Business DNA allowlisted claims; value-prop prompt lists **only** scorer-derived evidence lines.
5. **Bounded context:** DNA never dumps full files—`prompt_block` caps list lengths (e.g., pillars, claims).
6. **Repair context:** On failure, append **concrete violation strings** from programmatic validators, not vague “try again.”

## How outputs are validated

| Path | Mechanism |
|------|-----------|
| **Prospecting email** | `_validate_email`: three paragraphs; required “free for employer”; segment keywords (“cost of care” / “access”); first sentence must not contain “partnership”; merged banned terms (code + DNA voice). |
| **Prospecting questions** | `_validate_questions`: count ≥ `RULA_MIN_DISCOVERY_QUESTIONS` (config, default 3); each string ends with `?`. |
| **Parsing** | `_try_parse_email` / `_try_parse_questions`: `json.loads`, strip ``` fences if present. |
| **Value-prop explanation** | `_score_specificity` (0–100): needs multiple signal hits, comparative language, company name + source-bound words, lexical diversity; threshold **75** to accept LLM text. |
| **Committer** | `_parse_llm_json` to pydantic shape; merge rules vs `_heuristic_committer` regexes. |
| **`src/validators/response_validator.py`** | JSON schema-ish checks, semantic company-name check, DNA banned terms, **numeric claim allowlist** (`validate_claims`). **Primarily covered by tests**; not wired into `generator.py` hot path in the reviewed tree—treat as **shared library / future guardrail**. |

## Wrong, incomplete, or ambiguous model outputs

| Situation | Behavior |
|-----------|----------|
| **Provider error / empty text** | `GenerationResponse.ok` is false → prospecting uses **deterministic templates**; value-prop uses **template explanation**; committer uses **heuristics** or nulls. |
| **Invalid JSON / parse failure** | Treated as failure → same fallbacks as above. |
| **JSON valid but policy violations** | One **repair** call with **lower temperature (0.2)** and enumerated violations. If still invalid → template fallback (email/questions). |
| **Value-prop explanation too generic** | Specificity below 75 → **template** with deterministic signal strings. |
| **Committer ambiguous** | Prefer LLM when `name` present; else heuristic; weak pronoun + profile path documented in code. |
| **Cross-provider resilience** | `ModelRouter` retries the **same** `GenerationRequest` on the alternate provider if the first fails and keys exist. |
